# 🤖 Lucky-UI 组件库专属 AI 开发助手规范

> 本规范专为 **lucky-ui UniApp 组件库** 的迭代开发定制。
> 项目结构：`lucky-ui`（组件库本体） + `lucky-ui-test`（功能验证示例项目）

## 🧭 核心原则

### 1. 先理解，再动手
- **永远不要在完全理解需求和现有组件结构之前就开始写代码**
- 收到任务后，先用自然语言描述对任务的理解
- 如果需求模糊，**优先提问**，而不是猜测并开始实现
- 涉及 API 设计变更时，必须同时评估对示例项目的影响

### 2. 双项目联动原则
- **lucky-ui 的每一个改动，都要同步评估 lucky-ui-test 是否需要跟进**
- 新增组件 → 示例项目新增对应的使用页面
- 修改组件 Props/事件 → 示例项目同步更新调用方式
- 废弃 API → 示例项目移除或标注迁移方式

### 3. 最小化变更原则
- **只修改需要修改的部分**，不顺手"优化"无关组件
- 保持组件命名规范、目录结构、样式方案不变
- 每次迭代聚焦单一组件/功能，避免大范围重构

### 4. 真实性原则
- **绝对不伪造组件行为、Props 说明或测试结果**
- 对 UniApp 跨端差异不确定时，明确标注，不猜测
- 遇到知识边界坦诚告知，并提供验证方向

### 第三阶段：实现（IMPLEMENT）

**组件库（lucky-ui）开发规范：**

```
- Props 统一使用 LkProp 工厂方法定义（LkProp.string / LkProp.boolean / LkProp.enum 等）
- 枚举值统一定义为 const 对象 + 同名 type，与现有 props.ts 风格一致
- Emits 必须在 buttonEmits（或对应 xxxEmits）中显式声明，并提供校验函数
- 样式使用 scoped SCSS，通过 BEM mixin（@include b / e / m / when）组织
- 尺寸 / 颜色全部使用 CSS 变量 Token，不硬编码数值
- 跨端差异使用条件编译 #ifdef 处理，并注明平台差异原因
- 不使用 DOM API（保证小程序兼容性）
- 避免使用 UniApp 不支持的 CSS 属性
```

**示例项目（lucky-ui-test）开发规范：**

```
- 新增 Demo 页面前，必须先阅读至少一个现有 Demo 页面作为模板基准
- 严格复刻现有页面的布局结构、公共组件用法、间距视觉规范，不自创风格
- 示例代码真实可运行，不写假数据或占位代码
- 示例页面只做展示，不引入与组件无关的业务逻辑
- 新页面必须同步注册到 pages.json
- 新页面必须同步在首页导航列表追加入口 item
```

**代码注释风格：**
- 解释"为什么这样设计"，而不是描述"代码做了什么"
- Props 注释说明用途、可选值、默认行为（参考现有 buttonProps 注释风格）
- 跨端特殊处理必须注释说明平台差异原因

基于组件 Props 定义，Demo 页面按以下顺序分区展示（与 props.ts 枚举顺序对齐）：

3. 尺寸            → 遍历所有 size 枚举值
4. 形状            → 遍历所有 shape 枚举值（如有）
5. 状态            → disabled / loading / block 等布尔 Props 的 on/off 对比
6. 图标 / 插槽     → icon prop 或 slot 的使用示例（如有）
7. 综合示例        → 贴近真实业务的组合使用场景
```

> 每个分区的**标题文字、示例数量、代码注释**均对齐现有 Demo 页面风格，不多不少。

## 🏗️ 组件质量标准

### Props 设计
```
- 类型定义：统一使用 LkProp 工厂方法（string / boolean / number / enum 等）
- 枚举定义：const 对象 + 同名 type 双重导出，与 ButtonVariant 风格一致
- 默认值合理：组件开箱即用，无需强制传参
- 命名语义化：布尔值用 is / show / enable 等前缀
- 超过 6 个核心 Props 时，考虑是否职责过重，拆分子组件
- UniApp 平台能力 Props（openType 等）统一放在核心 Props 之后
```

### 事件设计
```
- 统一在 xxxEmits 对象中显式声明，并提供校验函数
- 事件名用 camelCase 声明，模板中用 kebab-case 监听
- 事件参数携带足够上下文（不只传 true/false）
- 表单类组件必须支持 v-model
- 平台特有事件（getuserinfo 等）单独归类并注释说明平台限制
```

### 样式设计
- 样式设计
  - 统一使用 BEM mixin：@include b(组件名) / @include e(元素) / @include m(修饰) / @include when(状态)
  - 组件内部状态变量用 --_xxx 命名（双横线+下划线前缀），与 button 样式保持一致
  - 尺寸 / 颜色全部使用 CSS 变量 Token（--lk-xxx），不硬编码
  - rpx 单位用于固定尺寸（如 border: 2rpx），百分比或 em 用于相对尺寸
  - 避免使用 * 通配符选择器
  - 点击态 / 激活态统一用 :active:not(.is-disabled):not(.is-loading) 保护
  - **跨端样式稳定性：在 `:style` 注入 CSS 变量时，若小程序端存在变量解析回退（渲染为默认值或空白），请为变量显式提供兜底值（如 `:style="{ '--_bg': bgColor || 'var(--lk-color-danger)' }"`），确保小程序环境渲染的一致性。**
```

### 跨端兼容
```
- 默认按最严格平台（微信小程序）编写
- 平台差异用 #ifdef 条件编译隔离，并注明原因
- 不依赖 window / document 等浏览器 API
- 测试矩阵：H5 + 微信小程序 + App（至少在迭代总结中提示需测试的平台）
```
