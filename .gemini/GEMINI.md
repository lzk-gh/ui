# 🤖 Lucky-UI 组件库专属 AI 开发助手规范

> 本规范专为 **lucky-ui UniApp 组件库** 的迭代开发定制。
> 项目结构：`lucky-ui`（组件库本体） + `lucky-ui-test`（功能验证示例项目）

---

## 🧭 核心原则

### 1. 先理解，再动手
- **永远不要在完全理解需求 and 现有组件结构之前就开始写代码**
- 收到任务后，先用自然语言描述对任务的理解
- 如果需求模糊，**优先提问**，而不是猜测并开始实现
- 涉及 API 设计变更时，必须同时评估对示例项目的影响

### 2. 双项目联动原则
- **lucky-ui 的每一个改动，都要同步评估 lucky-ui-test 是否需要跟进**
- 新增组件 → 示例项目新增对应的使用页面
- 修改组件 Props/事件 → 示例项目同步更新调用方式
- 废弃 API → 示例项目移除或标注迁移方式

### 3. 最小化变更原则
- **只修改需要修改的部分**，不顺手"优化"无关组件
- 保持组件命名规范、目录结构、样式方案不变
- 每次迭代聚焦单一组件/功能，避免大范围重构

### 4. 真实性原则
- **绝对不伪造组件行为、Props 说明或测试结果**
- 对 UniApp 跨端差异不确定时，明确标注，不猜测
- 遇到知识边界坦诚告知，并提供验证方向

## 🔄 标准工作流程

### 第一阶段：探索与理解（EXPLORE）

在触碰任何文件之前，必须完成：

```
1. 确认本次任务涉及哪个（些）组件
2. 阅读该组件现有源码，理解 Props / Emits / Slots 设计
3. 阅读该组件的 props.ts，理解类型、枚举、默认值定义方式
4. 查看示例项目中对应页面，理解当前 Demo 使用方式和页面结构
5. 识别是否有跨端兼容问题（H5 / 小程序 / App）
6. 确认是否影响已有的 Props/事件 API（破坏性变更？）
```

**内心独白示例（必须展示）：**
> 🔍 探索：先看 `lk-button` 的 `props.ts`...
> 💡 发现：`size` 当前支持 `sm / md / lg`，通过 `LkProp.enum` 定义，示例项目里有 3 处调用。
> 💡 发现：如果新增 `xs` 尺寸，不涉及破坏性变更，需同步更新 scss 中的 `@include m(xs)` 块。
> 🔍 探索：再看现有 Demo 页面结构，确认分区标题和 item 写法...
> 📋 计划：修改 props.ts → 修改 index.scss → 更新 Demo 示例页面新增 xs 展示...

### 第三阶段：实现（IMPLEMENT）

**组件库（lucky-ui）开发规范：**

```
- 必须使用 <script setup lang="ts"> + Composition API 组织代码
- 禁止使用 any，必要时使用 unknown 并进行类型收窄
- Props 统一使用 LkProp 工厂方法定义（LkProp.string / LkProp.boolean / LkProp.enum 等）
- 枚举值统一定义为 const 对象 + 同名 type，与现有 props.ts 风格一致
- Emits 必须在 buttonEmits（或对应 xxxEmits）中显式声明，并提供校验函数
- 样式使用 scoped SCSS，通过 BEM mixin（@include b / e / m / when）组织
- 严禁硬编码颜色，必须通过 CSS 变量 Token 实现（参考下方主题纪律）
- 跨端差异使用条件编译 #ifdef 处理，并显式标注 ⚠️可能存在平台差异 及原因
- 不使用 DOM API（保证小程序兼容性），禁止小程序不支持的 CSS 选择器
```

**示例项目（lucky-ui-test）开发规范：**

```
- 新增 Demo 页面前，必须先阅读至少一个现有 Demo 页面作为模板基准
- 严格复刻现有页面的布局结构、公共组件用法、间距视觉规范，不自创风格
- 示例代码真实可运行，不写假数据或占位代码
- 示例页面只做展示，不引入与组件无关的业务逻辑
- 新页面必须同步注册到 pages.json
- 新页面必须同步在首页导航列表追加入口 item
```

**代码注释风格：**
- 解释"为什么这样设计"，而不是描述"代码做了什么"
- Props 注释说明用途、可选值、默认行为（参考现有 buttonProps 注释风格）
- 跨端特殊处理必须注释说明平台差异原因，标记 `// ⚠️可能存在平台差异`

### 第四阶段：验证（VERIFY）

必须按以下流程逐步验证，不可跳步：

```
1. 静态检查：执行 pnpm run compat-check 确认跨端兼容性
2. 类型与规范：执行 eslint / vue-tsc 检查代码风格与类型安全
3. 逻辑复核：重读 props.ts / .vue / index.scss，确认定义与实现一致
4. 端侧预览：执行 build:h5 或 build:mp-weixin 查看实际渲染效果
5. 边界验证：模拟空 Props、极端尺寸、快速点击、嵌套等场景
6. 回归测试：按需执行 test:visual / test:mp，确保无视觉或逻辑回归
7. 给出手动验证步骤建议（在示例项目中如何验证）
```

## 🏗️ 组件质量标准

### 🎨 主题与 Token 纪律

- **禁止硬编码颜色**：严禁在样式中使用 `#hex`、`rgb/rgba` 等具体颜色值（除非用户明确要求固定色）。
- **组件 Token 优先**：优先使用组件专用变量（如 `--lk-switch-node-bg`），不要直接混用通用变量替代。
- **语义化背景/文本**：页面/容器背景优先用 `--lk-color-bg-page/container`，文本用 `--lk-color-text-*`。
- **亮暗适配**：改动颜色相关样式后，必须说明“亮/暗主题下预期变化”，并检查对比度。

### Props 设计
```
- 类型定义：统一使用 LkProp 工厂方法（string / boolean / number / enum 等）
- 枚举定义：const 对象 + 同名 type 双重导出，与 ButtonVariant 风格一致
- 默认值合理：组件开箱即用，无需强制传参
- 命名语义化：布尔值用 is / show / enable 等前缀
- 超过 6 个核心 Props 时，考虑是否职责过重，拆分子组件
```

### 事件设计
```
- 统一在 xxxEmits 对象中显式声明，并提供校验函数
- 事件名用 camelCase 声明，模板中用 kebab-case 监听
- 事件参数携带足够上下文（不只传 true/false）
- 表单类组件必须支持 v-model
```

### 样式设计
- **BEM 规范**：使用 @include b / e / m / when 组织样式。
- **私有变量**：组件内部状态变量用 `--_xxx`（双横杠下划线）前缀。
- **跨端稳定性**：在 `:style` 注入变量时必须提供兜底值，例如 `:style="{ '--_bg': bgColor || 'var(--lk-color-primary)' }"`。
- **点击态**：统一使用 `:active:not(.is-disabled):not(.is-loading)` 保护。

### 跨端兼容
- **最严标准**：默认按微信小程序标准编写。
- **平台隔离**：差异逻辑必须包裹在 `#ifdef` 中，并标注平台差异原因。
- **CSS 约束**：不依赖 window/document，避免使用小程序不支持的选择器。

## 🚀 Git 提交与推送规范

- **精准暂存**：若用户给出“文件清单”，仅 `git add` 清单内文件，严禁自动加入清单外改动。
- **分步执行**：在 PowerShell 环境下不要使用 `&&` 串联，应分步执行并检查每步结果。
- **提交信息**：简洁明了（1 行标题 + 可选 1-2 行原因），语义匹配（fix/feat/refactor）。
- **核对流程**：提交前后均执行 `git status --short` 进行复核。
- **失败处理**：`git push` 若因网络失败自动重试一次；持续失败则告知用户原因及建议。
- **回复要求**：明确列出本次暂存文件、commit hash 及 push 结果，说明已排除的文件。

## 📱 Demo 示例页面规范

> 严格复刻现有页面布局（如 pages/button/index.vue），严禁自创风格。

### 📐 内容组织规则
1. **基础用法** -> 快速上手
2. **变体/类型** -> 遍历 variant/type 枚举
3. **尺寸/形状** -> 遍历 size/shape 枚举
4. **状态** -> disabled/loading 等对比
5. **图标/插槽** -> icon/slot 示例
6. **综合示例** -> 贴近真实业务

### 🚪 入口同步
- **注册路由**：同步在 `pages.json` 中注册。
- **首页导航**：在 `pages/index/index.vue` 的对应分类下追加入口，保持字段格式一致。
